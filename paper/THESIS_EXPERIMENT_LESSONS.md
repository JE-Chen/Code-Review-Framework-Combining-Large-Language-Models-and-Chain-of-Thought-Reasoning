# 論文實驗與封存作業準則

本文件整理可跨專案使用的論文寫作、消融實驗、結果封存與交接原則。

## 證據與主張

1. 每個數字都要能回溯至逐案輸出、原始評分或可重現的彙整程式。
2. 人工黃金標準、模型產生參考資料、模型裁判與人工評分必須分開命名，不得互換。
3. 指標若使用不同分母或不同計數單位，不得共用傳統分類指標名稱。必要時使用
   「描述性調和平均」並說明限制。
4. 跨研究表格若資料、裁判或受評模型不同，只能作描述性對照，不可作因果效果比較。
5. 「未達統計顯著」表示目前資料不足以支持差異，不代表證明兩組相同。

## 實驗生命週期

每個實驗組必須明確標示下列其中一種狀態：

- `not_started`：尚未產生任何正式輸出。
- `in_progress`：生成或評分尚未全部完成。
- `complete`：生成、評分、彙整、統計與逐案追溯皆完成。
- `stopped_diagnostic_only`：已停止，只保留作錯誤或方法診斷，不得混入正式結果。
- `invalid`：控制條件失效，輸出不可用於研究結論。

進行中與停止診斷只能寫入研究資料狀態檔，不得提前寫入論文摘要、結果、貢獻或結論。

## RAG 消融設計

1. 服務顯示檢索已啟用，不代表文件已注入提示內容。每案須保存實際取回文件與數量。
2. 嵌入閾值不可直接跨嵌入模型沿用。更換模型後須重新稽核分數分布與命中率。
3. 無關語料可能在同一閾值下全部被過濾。若研究問題需要比較無關內容，應固定注入
   數量，或建立與相關組命中數相近的配對無關語料。
4. 除單一自變項外，模型權重、轉接器、權重精度、流程步驟、解碼方式、生成上限、
   案例順序及評分規準皆須保持一致。
5. 每個條件應保存機器可讀的條件清單，至少包含語料雜湊、閾值、注入方法、每案規則
   數、模型組態與生成上限。

## 遠端工作的逾時與取消

1. 用戶端逾時後，必須呼叫取消端點。
2. 取消後須輪詢至工作進入完成、失敗或取消等終止狀態，才能提交下一案。
3. 若無法確認前案終止，應中止整批，不可持續堆疊遠端工作。
4. 記錄提交時間、工作識別碼、取消結果、終止狀態與逐案耗時，供後續區分推論緩慢、
   排隊與用戶端逾時。

## 統計與判讀

1. 相同案例在兩個條件下的分數屬成對資料，應報告成對差異並使用成對檢定。
2. 分數尺度不宜假設常態時，可採雙尾 Wilcoxon 符號等級檢定。
3. 同時檢定多個維度時，須使用 Holm 等方法校正多重比較。
4. 除平均值外，應提供樣本標準差、逐案勝／同／負與實際 p 值。
5. 若只有部分案例實際接受處置，例如只有部分查詢命中規則，須在結果旁揭露，並限制
   因果解讀。

## DOCX 版本與語言

1. 原始 DOCX 永不就地修改，以唯讀來源產生下一個編號版本。
2. 所有修改由可重現的 `python-docx` 程式完成，修改前後均執行格式檢查及文字傾印。
3. 論文正文只保留方法、資料判讀與結論，不出現本機路徑、下載檔名、審查意見檔名、
   實驗目錄名稱或版本管理說明。
4. 使用正式繁體中文及臺灣常用資訊用語，避免簡體用詞、口語宣傳詞與代筆語氣。
5. 新增表格須在正文說明其目的、欄位、統計方法、結果與限制，並檢查欄寬未超出頁面。
6. 完成後確認來源檔雜湊未變、輸出可由 DOCX 函式庫開啟、壓縮結構完整且欄位設定可
   更新。

## Research Data 封存

完成的論文結果應同步封存下列資料：

- 原始案例與輸入提示。
- 每案生成輸出、實際檢索文件及評分結果。
- 聚合 JSON、可閱讀結果表與統計程式。
- 實驗條件清單、伺服器條件及執行日誌。
- 研究資料說明、結果表、審查意見對照與機器可讀狀態檔。
- 排除封存清單本身後，涵蓋所有檔案的 SHA-256 清單。

重建清單後須逐筆重算驗證，並掃描私鑰、存取金鑰、權杖、環境檔與其他敏感資訊。

## 交接紀律

1. 永久規則寫入代理指南，會變動的進度寫入獨立交接檔。
2. 交接檔須列出已完成結果、進行中批次、停止診斷、資料位置、重現入口與下一步順序。
3. 每次實驗狀態改變後立即更新交接檔，避免後續工作者依舊快照重複執行或提前下結論。
