[0072] <Heading 1> 第一章、緒論
[0073] <Heading 2 18.0pt> 1.1 研究背景
[0074] <Normal 14.0pt> 隨著大型語言模型（Large Language Models, LLMs）於自然語言與程式語言理解上的快速發展，程式碼生成與輔助開發已成為其在軟體工程領域之重要應用。在開發流程中，程式碼審查（Code Review）不僅是品質保證之關鍵環節，更是團隊協作與知識傳遞之核心機制：透過開發者間之相互檢視，潛在之邏輯錯誤、安全漏洞與效能問題得以在整合前被發現，開發風格與文化亦得以統一。傳統人工審查雖能有效提升品質與可維護性，卻需耗費可觀之人力與時間成本，於快速迭代之環境中漸成瓶頸。
[0075] <Normal 14.0pt> 以 LLM 輔助程式碼審查之研究背景可歸結為三個面向：其一，LLM 於語義理解上之突破，使其能跨越自然語言描述與程式結構之間的鴻溝，掌握程式碼之語義與設計意圖；其二，隨專案規模與協作複雜度上升，確保程式碼一致性與可維護性之需求日益迫切，尤以大型協作專案為甚；其三，現有靜態分析與 Lint 工具雖能檢出部分錯誤，於語義層級之理解與設計合理性判斷上仍顯不足，這正是 LLM 可補足之空間。
[0078] <Heading 2 18.0pt> 1.2 研究動機
[0079] <Normal 14.0pt> 在軟體開發流程中，人工程式碼審查雖為品質保證之核心，卻隨專案規模龐大、程式碼複雜度上升而面臨效率不足、審查標準不一致、審查者主觀性過高等問題，於大型專案中漸成開發瓶頸，亟需更具智慧化與自動化之輔助工具。大型語言模型之出現帶來契機：其能同時理解自然語言與程式語言，於程式碼結構、語義、風格一致性與潛在錯誤之檢測上提供支持，並較傳統靜態分析工具更能捕捉設計意圖、提出符合工程規範之改進建議。
[0080] <Normal 14.0pt> 然而，若把 LLM 直接放上審查崗位，會浮現三個彼此牽連之問題，這三者正是本研究設計之起點。其一為幻覺：模型可能編造看似合理卻不正確之內容，例如指出一個並不存在之函式呼叫，使審查意見不可信。其二為輸出不穩定：相同的程式碼變更，模型於不同次或不同提示詞下可能給出不一致之結論，使審查結果難以複現與比較。其三為缺乏領域規範：模型只見過泛用語料，並不知道「你這個專案」的命名慣例、安全要求或設計準則，因而給出與專案脈絡脫節之建議。
[0081] <Normal 14.0pt> 這三個問題並非各自獨立，而是構成一條因果鏈，需要一套整合設計而非單點技巧來回應。把整段審查塞進一個提示詞，正是輸出不穩定與幻覺的溫床：模型在一次推理中要兼顧摘要、風格、邏輯與安全，注意力分散而易遺漏或臆造；缺乏領域規範則使它更傾向以「一般而言」之套話填補，加重幻覺。因此本研究主張，唯有同時做到「把審查拆細、逐步可查」「把專案規範即時餵給模型」「把這套能力穩定保留於可部署之模型」，審查結果才能同時兼具一致性、可解釋性與可維護性，並使 LLM 與人類審查者形成互補。此即本研究之核心動機，下節之研究問題與隨後之研究目的即由此而生。
[0082] <Heading 2 14.0pt> 1.3 研究問題
[0083] <Normal (Web) 14.0pt> 本研究根據前述研究背景與動機，將自動化程式碼審查框架之設計與評估聚焦於下列四個研究問題：
[0084] <Normal (Web) 14.0pt> RQ1：以多階段思維鏈提示詞拆解審查任務、並結合 LoRA 微調之框架，相較傳統單一提示詞與 CRSCORE++ 等既有評估方法，能否於完整性、簡潔性與相關性等面向顯著提升審查品質？
[0085] <Normal (Web) 14.0pt> RQ2：在固定參數規模下，僅引入多階段提示詞而不進行模型微調，是否即可帶來品質提升？以分離提示詞工程與權重微調二者之獨立貢獻。
[0086] <Normal (Web) 14.0pt> RQ3：多階段提示詞與模型微調二者，對最終分數之邊際貢獻分別為何？哪一項為品質改善之主要因素？
[0087] <Normal (Web) 14.0pt> RQ4：自動化 LLM-as-a-Judge 評分之結論，是否能由獨立人工評分交叉驗證、並於可維護性與正確性等深層語義維度上保持一致？
[0088] <Heading 2 18.0pt> 1.4 研究目的
[0089] <Normal 14.0pt> 本研究之目的，在於回應上節所述 LLM 審查之三大問題——幻覺、輸出不穩定、缺乏領域規範——並驗證一套整合框架能否使審查結果同時具備一致性、可解釋性與可維護性。所謂目的，指本研究「要解決什麼」，與下節之貢獻「實際做出了什麼」分屬兩事，特先界定以免混淆。
[0090] <Normal 14.0pt> 對應地，本研究將三個問題明確映射至三項設計，作為全篇之論證主軸（於第三章設計、於第五章佐證）：其一，針對輸出不穩定與一致性差，以多階段思維鏈將審查拆解為摘要生成、初步審查、靜態分析、程式碼異味偵測與最終彙整等循序步驟，每步僅承擔單一子任務、產物逐一落為 .md 檔，並以統一之全域規則前綴貫串，使審查可逐步追溯而趨於穩定；其二，針對缺乏領域規範，以 RAG 於每次審查前檢索相關專案規則（餘弦相似度達門檻者）動態注入為條件式之全域規則，使模型不重新訓練即可按專案規範審查，貼合領域且可維護；其三，針對幻覺與落地成本，以知識蒸餾結合 QLoRA 把大型教師模型之審查能力輕量化移轉至學生模型，使逐步且有
[0091] <Normal 14.0pt> 為釐清整合框架內部各元件之相對作用，本研究進一步以消融實驗分離「多階段提示詞流程設計」與「模型權重微調」二者對審查品質之獨立貢獻。需強調者，此一分離旨在理解框架之組成而非主張「以一者取代另一者」：三項設計各自回應不同問題，缺一即難同時達成一致性、可解釋性與可維護性，其量化證據與彼此互補之關係，將於第五章結果分析中回填。

[0104] <Heading 1> 第二章、文獻探討
[0105] <Heading 2 20.0pt> 2.1 程式碼審查
[0106] <Normal 14.0pt> 程式碼審查是軟體開發中極為重要的一環，目的在確保程式碼之品質、可維護性與一致性。它不只是找出錯誤或漏洞，更是一種知識分享與團隊協作的方式：透過審查，潛在之邏輯錯誤、安全性漏洞、效能瓶頸，乃至命名不清或結構不佳之程式碼得以在合併前被攔截，大幅降低後續維護成本。Kakimoto 等人指出程式碼審查可在整合測試前發現 50% 至 70% 的缺陷 [13]，避免上線後才付出更高代價修正，審查亦能促進成員交流，使不同開發者理解彼此之設計脈絡並統一團隊風格。
[0107] <Normal 14.0pt> 對本研究的意義：人工審查所提供之「品質把關 + 知識傳遞」雙重價值，正是自動化審查必須盡量保留之目標。本研究之多階段流程刻意逐步輸出可讀之中間文件（摘要、初審、靜態分析、異味），即是為了讓自動審查不只給出結論，更能像人類審查般呈現推理脈絡，維持可解釋與知識傳遞之價值。
[0108] <Heading 2 20.0pt> 2.2 靜態分析工具 （STAs）
[0109] <Normal 14.0pt> 靜態分析工具（Static Analysis Tools, STAs，於程式未執行前即掃描原始碼以找出問題之工具）透過對原始碼或編譯後中介表示進行系統化掃描，找出潛在錯誤、風格問題、安全漏洞與不符最佳實踐之處。與需實際執行程式之動態測試不同，靜態分析能於開發早期就提供回饋，降低後期修正成本，常見如 SonarQube、Coverity、ESLint、Pylint，並常整合於 CI/CD 流程。然而其檢測能力受限於規則僵化：規則本身難以涵蓋設計合理性、命名語義等高層次判斷，規則更新亦不易，故相當比例之問題無法被涵蓋。
[0110] <Normal 14.0pt> 對本研究的意義：STAs 之「規則明確、可重現」與其「無法理解語義與設計意圖」恰成一體兩面。本研究因此不以 LLM 取代靜態分析，而是在多階段流程中保留 Linter 步驟作為確定性基礎，再由後續之 CoT 推理步驟補上 STAs 難及之語義層判斷，使二者互補。
[0111] <Heading 2 20.0pt> 2.3 大語言模型 （LLM）
[0112] <Normal 14.0pt> 大語言模型（Large Language Model, LLM）是一種基於深度學習之人工智慧系統，核心為以神經網路處理與生成自然語言，通常採 Transformer（一種以自注意力機制為核心之網路）架構。透過大量文本之預訓練，模型學得語言之統計規律與上下文邏輯，再於微調階段針對特定任務（如問答、翻譯、程式碼生成）調整。現代模型參數量常達數百億甚至上千億，使其能捕捉極複雜之語義關聯，惟亦因此面臨偏見與幻覺等問題，需搭配人工監督。
[0113] <Normal 14.0pt> 對本研究的意義：LLM 強大的語義理解力，是本研究能跨越「自然語言描述」與「程式結構」鴻溝之前提，但其幻覺與不穩定，也正是 §1.2 三問題之來源。本研究後續以 RAG（補領域知識）與多階段流程（穩定輸出）對症處理，即建立在對 LLM 此一「能力與風險並存」本質之認識上。
[0114] <Heading 2 20.0pt> 2.4 LLM-as-a-Judge
[0115] <Normal 14.0pt> LLM-as-a-Judge 是一種利用大型語言模型作為「裁判」來自動評估其他模型輸出的新興方法，它的核心思想是讓強大的 LLM 依照指定標準對文本生成結果進行打分、排序或比較，以取代或輔助人工標註，如同Kakimoto研究的結果LLM 的錯誤偵測能力與人類審查者相當，統計上無顯著差異，LLM 在程式碼審查中可作為輔助工具，提供初步檢查與提示，但尚不足以取代人類[13]。
[0116] <Normal 14.0pt> LLM-as-a-Judge 的出現源於傳統評估方法的不足：過去常用的基於匹配或詞嵌入的指標（如 BLEU、ROUGE）難以捕捉語言的細微差異，無法全面衡量生成文本的質量。隨著 GPT-5 等模型的強大理解能力被驗證，研究者開始嘗試讓 LLM 直接充當「評審」，透過提示詞（prompt）定義評估標準，並讓模型依此對輸出進行判斷。這種方法在多種場景中展現出高效性。
[0117] <Normal 14.0pt> 在實際應用中，LLM-as-a-Judge 可以採用兩種主要方式：一是成對比較，即讓模型同時看到兩個候選輸出並判斷哪一個更好；二是單一評分，即對單一輸出依據正確性、相關性、偏見程度、是否存在幻覺等維度給予分數或標籤。這樣的評估方式更接近人工標註，能提供描述性判斷而不僅僅是數值指標。代表性工具如 AlpacaEval 就是基於此，利用 GPT-4 Turbo 對模型輸出進行自動化比較，並已廣泛應用於 LMSYS Chatbot Arena 等排行榜。
[0118] <Normal 14.0pt> 然而，LLM-as-a-Judge 也存在挑戰：偏見與一致性問題（不同模型可能給出不同判斷）、成本問題（使用高性能 LLM 進行大規模評估需要大量算力與費用）、以及 可靠性問題（LLM 的判斷仍可能出現幻覺或錯誤）。因此，研究者正在探索如何設計更穩健的提示詞、建立基準數據集，以及結合人工與自動化方法來提升可信度，如Naik等人在他們的論文「CRSCORE」裡定義了三個維度，簡潔性、完整性、相關性來讓LLM進行評分[7]，且Kapadnis等人的「CRSCORE++」依然沿用了三個維度與LLM評分制度[10]，Crupi及Tufano指出傳統自動化評估方式（如 BLEU、ROUGE、METEOR
[0119] <Normal 14.0pt> 對本研究的意義：LLM-as-a-Judge（讓強大之 LLM 依指定標準替其他模型輸出打分之方法）使本研究得以在不依賴大量人工標註之前提下，對「多階段 vs 單一提示詞」等不同設定之審查品質進行可重現之比較。其已知之偏見與一致性風險，則促使本研究以雙評審模型（GPT-5、Gemini-3）加人工評分交叉驗證——此一安排即為第四章評估設計與第五章交叉驗證之文獻依據。
[0120] <Heading 2 20.0pt> 2.5 RAG
[0121] <Normal 14.0pt> 檢索增強生成（Retrieval-Augmented Generation, RAG）是一種結合「檢索」與「生成」之技術：其直覺是，模型本身不知道你專案的規範，RAG 就在每次生成前，臨時把最相關的幾筆外部資料（如專案規則）找出來塞進提示詞，讓模型照這些資料作答，而不必為此重新訓練模型。具體分兩階段：檢索階段依查詢自外部知識庫（如向量資料庫）找出最相關之片段，生成階段再將片段與原始查詢一併輸入以生成回答。因模型不再單靠內部記憶、而能引用真實外部來源，RAG 能有效降低幻覺並提升準確性，目前已應用於安全建議與開發輔助系統。
[0122] <Normal 14.0pt> 對本研究的意義：RAG 正是本研究回應「缺乏領域規範」此一問題之核心機制。本研究以 RAG 將手寫之專案規則於審查前動態注入為條件式之全域規則（詳見 §3.1.4 與 §3.4），使審查貼合專案脈絡且可隨規則文件更新即時調整，而無需每次重新訓練模型——這是「可維護性」性質之來源。
[0123] <Heading 2 20.0pt> 2.6 PEFT
[0124] <Normal 14.0pt> 參數高效微調（Parameter-Efficient Fine-Tuning, PEFT）是一類針對大型語言模型之微調技術，核心理念為只調整模型中一小部分參數，而非如傳統微調更新全部權重。由於大型模型常擁數十億至上百億參數，全參數微調之運算與儲存成本極高，PEFT 透過引入輕量模組（如 LoRA、Prefix Tuning、Adapter）將可訓練參數限制於少量附加層，使模型在保留原有能力之同時快速適應特定任務，大幅降低訓練成本與記憶體需求。其中 LoRA（Low-Rank Adaptation，低秩適應）以低秩矩陣近似全參數更新，在極少參數下仍可逼近完整微調之效果，並能如插件般模組化載入同一
[0125] <Normal 14.0pt> 對本研究的意義：PEFT / LoRA 是本研究「在有限 GPU 資源下完成 30B 級模型微調」之關鍵前提。本研究以 LoRA 適配器保留可拆卸性（未與基座合併），使同一基座得掛載多組任務適配器並動態切換——這對知識蒸餾後之輕量化部署，以及未來擴充，皆提供彈性。
[0126] <Normal 14.0pt> LoRA 是一種近年在人工智慧領域非常重要的輕量化微調技術。它的核心理念是：在不需要重新訓練整個大型模型的情況下，透過引入低秩矩陣來讓模型快速學習新的概念、角色或風格。因為完整訓練模型的成本極高，而 LoRA 只需訓練少量參數即可完成微調。後來，LoRA 被廣泛應用到影像生成領域，尤其是 Stable Diffusion 等模型，成為一種模組化外掛：使用者可以針對特定角色、畫風或藝術風格訓練出小型的 LoRA 模型，然後將它注入基礎模型中，就能生成符合需求的作品。
在大型語言模型的訓練中，LoRA 扮演了非常關鍵的角色，因為它能大幅降低微調的成本與資源消耗。傳統的全參數微調（Full Fine
[0127] <Normal 14.0pt> 在實際應用上，LoRA 通常與 Hugging Face 的 PEFT 框架結合使用。開發者只需在原本的訓練流程中加入 LoRA 的設定，就能快速將模型轉換為可微調的版本，並在有限資源下完成訓練。例如，對一個 7B 參數量的模型，使用 LoRA 可以在單張顯示卡上完成特定任務的微調，而不需要昂貴的伺服器或多 GPU 集群。
[0128] <Normal 14.0pt> 更進一步，LoRA 的設計也讓它能以模組化的方式存在：不同的 LoRA 模型可以像插件一樣被載入到同一個基礎模型中，讓 LLM 在保持原有能力的同時，快速學會新的任務或風格。這種靈活性使得 LoRA 成為目前 LLM 訓練與部署的主流方法之一。
[0129] <Heading 2 20.0pt> 2.8 QLoRA
[0130] <Normal 14.0pt> QLoRA（Quantized Low-Rank Adaptation，量化低秩適應）是專為大型語言模型設計之高效微調技術，結合 LoRA 與量化（Quantization，以較低位元表示權重以省記憶體）之思想，目的是在顯著降低硬體資源消耗之同時，仍維持接近全參數微調之效果。傳統大模型微調需巨量記憶體（例如 65B 模型於 float16 下可能需逾 130GB），多數研究者難以負擔。QLoRA 之核心創新在於將預訓練權重量化為 4-bit 精度（特別是 NF4，Normal Float 4-bit），再以雙重量化進一步壓縮記憶體，並輔以分頁優化器處理記憶體尖峰，使大型模型得於單機有限資源下完
[0131] <Normal 14.0pt> 對本研究的意義：QLoRA 是本研究訓練端之具體實作選擇。本研究於訓練時以 4-bit NF4 加雙重量化降低顯存佔用，推論時則改以 bf16 載入（因 A3B MoE 架構於 4-bit 推論路徑上會 densify 而導致記憶體不足），此一「訓練量化、推論不量化」之取捨，使輕量化得以在不犧牲推論穩定性之前提下落地。
[0132] <Heading 2 20.0pt> 2.9 知識蒸餾（Knowledge Distillation）
[0133] <Normal 14.0pt> 大型語言模型的知識蒸餾是一種將大型教師模型（teacher model）的能力轉移至較小學生模型（student model）的訓練方法，其核心目的在於讓學生模型在參數量與運算成本較低的情況下，盡可能學習教師模型的語言理解、生成品質與推理風格。此概念最早由 Geoffrey Hinton 等人提出[22]。 在此方法中，學生模型的訓練目標等同於一般監督式學習（Supervised Fine-Tuning, SFT）：首先以高效能教師模型（例如 GPT-5 或 Claude）對大量文字或指令資料產生高品質回應，接著將這些回應視為標準答案，讓學生模型透過微調更新參數，使其輸出結果盡可能接近教師模型
[0134] <Normal 14.0pt> 學生模型可透過大量由教師模型生成之問答資料、推理範例與對話樣本進行訓練，藉由模仿教師之完整回應內容與表達方式，於語氣、段落結構與解題流程上逐步貼近教師。此處最直觀之類比是：知識蒸餾就像讓小模型當學徒，照著大模型寫好之範例與推理過程學會審查。透過足夠規模與多樣性之教師資料，學生模型可在實務表現上逼近教師之品質，並可反覆生成新資料、重新訓練以形成持續改進之蒸餾流程。
[0135] <Normal 14.0pt> 對本研究的意義：知識蒸餾是本研究把「大型教師之高品質審查能力」搬到「可部署之小模型」之橋樑。本研究以具 CoT 之少樣本提示向教師大模型蒸餾出帶推理軌跡之資料（而非僅有結論），使學生模型學到的是「如何逐步推理地審查」，而非死記答案——這正是審查可解釋性得以被保留之原因。

[0144] <Normal (Web) 18.0pt> 3.1.1 進入點與持續整合層（Entry & CI）
[0145] <Normal 14.0pt> 審查由 GitHub PR 觸發，經 GitHub Actions 工作流程啟動，工作流程以 actions/cache 保存跨次執行之狀態與差分快取，並透過 prthinker 命令列介面（CLI）作為 runner 進入點，於本機路徑時亦可直接呼叫 cot.py 執行五階段審查。

[0147] <Normal 14.0pt> 對外推論服務以 Cloudflare 提供傳輸層安全（TLS）終端，經 nginx 反向代理進行路由，導向自架之 FastAPI 伺服器，為因應 30B 級模型單次推論可能超過反向代理之 idle timeout，伺服層提供非同步 job 端點（submit / poll），並於啟動與執行期設置防護（boot 與 runtime guards）。

[0350] <Normal 14.0pt> ‧ 遺留衝突標記（merge_markers）：偵測新增行之 <<<<<<< / >>>>>>> / diff3 |||||||（略過 ======= 分隔線以免誤判 Markdown 底線），因遺留標記幾必為衝突解析失誤。

[0354] <Normal 14.0pt> ‧ 大塊新增（large_hunk）：量測每檔最長連續新增行，逾門檻（預設 80 行）即標示，使單筆大量貼上或生成內容被刻意決定「略讀或細讀」，而非誤認為分散之手寫小修改。

[0356] <Normal 14.0pt> ‧ 刪除檔（deleted_files）：列出整檔刪除，使被移除之測試或安全防護不致湮沒於大量刪除行中。
[0357] <Normal 14.0pt> ‧ 重命名／移動（rename_map）：自 diff 之 rename from / rename to（含 similarity index 相似度）解析，使純移動不被重複審為「新檔加刪檔」。
[0358] <Normal 14.0pt> ‧ 低關注度噪音檔（noise_files）：將 lock 檔、minified／生成 bundle、 vendored 第三方樹與提交之快照分類為「可略讀」之提示；僅屬建議，不丟棄檔亦不左右最終結論。
[0359] <Normal 14.0pt> ‧ 新增延遲標記（new_markers）：僅就新增行掃描 TODO / FIXME / XXX / HACK / BUG，逐項列出 path:line，使本次新引入之技術債於提交時即可見；context 行上之既有標記不計入。

[0365] <Normal (Web) 20.0pt> 4.1 基準資料
[0366] <Normal 14.0pt> 直覺先行：要公平地評估審查框架，測試題目本身不能偏心——既不能全由人手寫而沾上個人風格，也不能全由單一模型生成而讓受測者「考自己擅長之題」。本研究因此以 GPT-5 與 Copilot 共同產生 44 筆測試資料集，並經人工驗證，兼顧資料品質、任務覆蓋率、模型泛化能力與評估穩定性，用以驗證整個流程（摘要、code review、linter 分析、code smell 判斷與最終總結）於不同類型程式碼情境下是否穩定、準確且一致。以兩種不同生成風格與訓練背景之工具共同產生資料，可有效擴展語法風格、命名習慣、設計模式與錯誤型態之分佈，避免單一來源造成之風格偏差與分佈集中。
[0367] <Normal 14.0pt> 我們產生了 44 筆測試資料，在實務上，小於 20 筆的資料集通常難以覆蓋多維度錯誤類型，例如效能問題、安全漏洞、命名不一致、過度耦合、重複邏輯、異常處理缺失等；而超過上百筆又會增加人工驗證與標註成本，使迭代效率下降。
[0368] <Normal 14.0pt> 這 44 筆高品質測試資料也可以作為教師模型與學生模型之間的對照樣本，用於比較不同模型在相同輸入下的輸出差異，透過 GPT-5 與 Copilot 共同產生並經人工驗證的資料集，可以確保測試資料既多樣又具代表性，避免模型僅在單一分佈下表現優秀。

[0495] <Normal 14.0pt> §3.7.2 所述之 MCP 整合層使本框架可在 IDE 內直接觸發審查，後續可比較 IDE 觸發（push 前）與 CI 觸發（push 後）兩種時機對開發者接受率與後續修正成本之差異。在 ops 補強面，可將框架隨附之 SQLite cache / telemetry 遷移至 Redis 與 PostgreSQL 以擴及多 server 共享之企業環境，並補上 drift watcher：以固定之 golden PR 集合定期重跑審查，比對輸出相似度，一旦偏離既有 baseline 即觸發告警。
[0496] <Normal (Web) 18.0pt> 6.4.5 §3.8 所述十七項研究級擴充機制之實證評估
[0497] <Normal 14.0pt> §3.8 所述十七項機制目前僅完成框架實作；其端到端品質效益需後續以真實 PR 流量驗證。為避免日後補實驗時設計分歧，本節為各機制標示最小可驗證實驗骨架；所列指標皆為公開可重現之量度，避免引入新主觀量表。其中 （a）–（m） 對應 §3.8.1–§3.8.13、（n）–（q） 對應 §3.8.15–§3.8.18；
[0498] <Normal 14.0pt> (r)–（t） 分別對應 §3.8.19 之可觀測層、§3.8.22 之審查導覽與 §3.8.23 之無模型定向信號（三者均屬部署 / 輸出層，不計入十七項，列此便於對照）。
[0499] <Normal 14.0pt> (a) Prompt-injection robustness（§3.8.1）：擴充 seed.jsonl 至每攻擊類別 ≥ 30 例，於四類後端各跑一遍，以 SQLite 表格內之 bypassed / detected 欄聚合為 detection rate 與 false-alarm rate 之偏序對照。
[0500] <Normal 14.0pt> (b) Closed-loop 多輪對話（§3.8.2）：於同一 PR 連續推 ≥ 5 次提交，比較啟用 --reply-to-author 與否之 round-k 重複 finding 比率與作者採納率。
[0501] <Normal 14.0pt> (c) Counterfactual 審查（§3.8.3）：抽 ≥ 50 個 design-choice 類 finding，請 ≥ 3 名人工審查者就「呈現替代方案是否影響其最終決策」之 Likert 5 點評分作為效益指標。
[0502] <Normal 14.0pt> (d) Provenance 稽核（§3.8.4）：以人工標記 ≥ 100 條 finding 之「正確 / 誤判」標籤，比較有引用 vs 無引用兩組之 precision，並用 confidence 與真實正確率之 ROC AUC 量化自評之校準度。
[0503] <Normal 14.0pt> (e) Force-push 差分（§3.8.5）：於連續 30 天之 PR 流量上比較啟用 --diff-since-last 與否之 token 用量、cache hit 比率與 false-reuse 比率（cache hit 但模型若實際重跑會產出不同 finding 之比例）。
[0504] <Normal 14.0pt> (f) Suggestion sandbox 驗證（§3.8.6）：以 ≥ 100 條 suggestion 在 sandbox 內套用後跑 pytest -x，計算 pass / fail / skip / error 四類比例；另以人工標記真實正確性，計算 sandbox 之 verdict 與人工判斷之 Cohen's κ。
[0505] <Normal 14.0pt> (g) Cross-language API drift（§3.8.7）：構造 ≥ 30 個 mixed-language PR（後端 rename、欄位刪除、type 變更）作為 ground-truth，計算 precision / recall。
[0506] <Normal 14.0pt> (h) PR 類型自適應（§3.8.8）：在 ≥ 200 個公開 PR 上以 commit msg prefix / labels 為 ground-truth 計算分類 accuracy / macro-F1；並比較啟用 --pr-classify 前後之每類 PR finding 精確率。
[0507] <Normal 14.0pt> (i) Reproducibility 訊號（§3.8.9）：對固定 PR 集合各跑 5 trials，以兩兩之 （path, line, normalised-comment） 重合率作為內部一致性指標；驗證 stable 標記與真實正確率之相關性。
[0508] <Normal 14.0pt> (j) Dependency upgrade impact（§3.8.10）：以公開 advisory（GHSA / CVE）作為 ground-truth breaking change 之來源，於 ≥ 50 個歷史 dependency bump PR 上計算 precision / recall。
[0509] <Normal 14.0pt> (k) Reviewer personas + conflict surfacing（§3.8.11）：於 ≥ 50 個 design-heavy PR 上比較單 lens 與 personas（含 conflict step）兩設定下，人類審查者「需介入決策」之留言數與最終 PR 之 revert 率。
[0510] <Normal 14.0pt> (l) Risk-weighted attention（§3.8.12）：以歷史 bug-fix PR 之檔案分布為 ground-truth，建立 risk score 與「該檔於下一季出現 bug fix commit 之機率」之相關係數；並對權重 （0.4, 0.3, 0.3） 進行敏感度分析。
[0511] <Normal 14.0pt> (m) Diff entropy（§3.8.13）：以公開「PR 被拆」事件作為 ground- truth，計算 verdict ∈ {focused, wide, bomb} 與「該 PR 後續被拆」之關聯。
[0512] <Normal 14.0pt> (n) Active-learning derived lessons（§3.8.15）：先令 dismissed / accepted 累積至雙語料各 ≥ 100 筆後，於同一份固定 PR 集合上以 paired bootstrap 比較啟用 --lessons 與否之 inline finding precision 與作者再次按 👎 比率。輔以時間切片：每週重跑 derive-lessons，比較其輸出之新規則與既有規則之 Jaccard 重疊，量化規則庫之收斂速度。
[0513] <Normal 14.0pt> (o) Cross-PR finding clustering（§3.8.16）：於 ≥ 500 筆累積 finding 上以 discover-rules 跑全 grid 之 --similarity-threshold ∈ {0.75, 0.80, 0.85, 0.90} × --min-cluster-size ∈ {3, 5, 10}；以 ≥ 3 名人工審查者就「此 cluster 是否確為一條值得寫入規則庫之真實重複規則」之 Likert 5 點評分作為品質指標，並計算評審間 Cohen's κ。並以「成為框架候選 → 被人類採納為 rules-dir 條目」之漏斗比率作為實用性指標
[0514] <Normal 14.0pt> (p) Repo knowledge graph grounding（§3.8.17）：以 ≥ 30 個跨語言大 repo 為樣本，於每 repo 上各跑啟用與未啟用 --kg-ground 之全 PR 集合。以人工標記之「該 finding 提及之符號是否實際存在於 repo」作為 ground-truth，計算「符號虛構率」之相對降幅；輔以 --kg-ground 之 prompt token 開銷與每 PR wall-clock 之 trade- off 表。視覺化模組之效益另以團隊內部使用次數 / page-view 等工程量度報告，不混入研究主張。
[0515] <Normal 14.0pt> (q) Crash-safe partial review（§3.8.18）：刻意於 CI 矩陣中注入 N 次中斷（concurrency: cancel-in-progress 觸發、人工 cancel、模擬 GPU OOM 之 ask/cancel 連發），以 --incremental-save-dir 啟用前後各跑 ≥ 50 次中斷實驗，比較「中斷後可恢復之 per-file 完成數量」之中位數與「未恢復則需重跑之 GPU-second 總開銷」之減幅。本項屬可靠性工程量化評估，與品質研究分流。
[0516] <Normal 14.0pt> （r） 推論伺服器指標端點與監控疊加層（§3.8.19）：以固定之合成負載（固定 PR 集合連續送審）量測啟用 /metrics instrumentation 前後之每請求延遲差與伺服器吞吐差，作為觀測開銷之上界；並以 ≥ 50 次注入之故障情境（GPU OOM、後端重啟、502）量測「自告警觸發至營運者定位根因」之時間，比較有無 Grafana 儀表板兩組之 MTTR（平均修復時間， mean time to repair）。本項屬可觀測性工程量化評估，與品質研究分流。
[0517] <Normal 14.0pt> （s） 審查導覽與分流輔助（§3.8.22）：就逐檔變更走查，以 ≥ 50 個 PR 令 ≥ 3 名審查者就「走查是否縮短理解該檔變更之時間、是否準確反映 diff 而無臆造」之 Likert 5 點評分與事實性誤判率作為效益與安全指標；另以開啟與未開啟導覽輔助兩組量測「首條留言前之導覽時間」與缺陷遺漏率之差。本項屬輸出 / 呈現層量化評估，與核心品質研究分流，不計入十七項。
[0518] <Normal 14.0pt> （t） 無模型定向信號（§3.8.23）：以 ≥ 50 個含已知植入樣態之 PR（人工注入遺留衝突標記、Trojan-Source 字元、遺留除錯敘述、吞噬例外等）為正樣本，另以等量未植入之乾淨 PR 為負樣本，量測十二項信號之逐類召回率與誤報率；並以開啟與未開啟無模型 triage 兩組，量測「於不呼叫模型之前提下，確定性信號先行標示之高風險檔佔最終人工確認問題之比例」與「triage 階段之 wall-clock （因零推論，預期為毫秒級）」。本項屬輸出 / 呈現層之確定性偵測量化評估，與核心品質研究分流，不計入十七項。
[0519] <Normal 14.0pt> 上列十七組研究級實驗與 （r） 之可觀測層工程實驗、（s） 之導覽層實驗、（t） 之無模型信號實驗皆需累積實際語料；本論文之主要貢獻仍為 §5.1 / §5.2 所述之多階段 CoT + LoRA 微調 + RAG 之整合設計與驗證，§3.8 與 §6.4.5 之內容明示為框架設計貢獻與後續工作之承接介面，不影響本論文之核心主張。
