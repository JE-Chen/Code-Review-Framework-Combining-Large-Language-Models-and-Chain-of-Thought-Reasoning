### Abstract

- **論證**：一段式 problem → approach → key result → significance。
- **材料**：人工審查瓶頸；多階段 CoT + RAG + KD/QLoRA；CRSCORE++ 0.86/0.83/0.64 優於基準。

### §1 Introduction

- **1.1 背景**：LLM 於 SE 的滲透 + code review 在開發流程中的樞紐地位（品質/協作/知識傳遞）。
- **1.2 動機**：人工審查的三重瓶頸（時間成本線性增長 \[4\]、標準不一、資深負擔 \[5\]）→ 傳統 STA/Lint 的語義侷限 \[1\]\[6\] → LLM 的潛力**與**三個未解問題（幻覺、不穩定、缺領域規範）。
- **1.3 問題與目的**：如何使 LLM 審查同時具備一致性 / 可解釋性 / 可維護性。
- **1.4 貢獻**：(1) 多階段 CoT 審查流程設計 (2) KD+QLoRA 輕量化 (3) RAG 規則注入 (4) LLM-as-Judge-Our 五維評估；末句一句帶過隨附框架（不展開、不附版本號）。
- **1.5 論文架構**。
- **[深]** 1.2 把「現有不足 → 為何需要本方法」鋪成一條因果鏈，而非並列羅列缺點。**[銜]** 1.2 結尾的「三個未解問題」直接成為 1.3 的問題定義、1.4 的貢獻逐一回應，形成首尾呼應。

### §2 Related Work

- **論證**：四條技術線**收束到一個研究缺口**，而非逐篇流水帳。
  - (a) code review + STAs：規則僵化、語義盲區 \[1\]\[6\]\[12\]。
  - (b) LLM + LLM-as-Judge：審查潛力 \[2\]\[3\] + 評估可與人類相當 \[13\]、補 BLEU/ROUGE 之不足 \[11\]。
  - (c) RAG：降幻覺、注入領域知識 \[8\]\[19\]。
  - (d) PEFT/LoRA/QLoRA + KD：有限資源下保留能力 \[16\]\[17\]\[21\]\[22\]。
- **2.x 研究缺口**：現有多採單一提示詞或受限規則，於**語義理解、跨階段一致性、評估可信度**三處留缺口 → 本研究以多階段 CoT + RAG + 雙重 Judge 補足。
- **[構]** 每條技術線結尾都顯式指向「本研究如何用它」。**[深]** 缺口不是泛談，而是三個具體維度，且每個維度對應 §3/§4 的一個設計。

### §3 Methodology

- **3.1 KD + QLoRA 訓練管線**：教師（高推理大模型）→ CoT 提示蒸餾 → 資料清洗/格式化 → QLoRA 微調學生（Qwen3-Coder-30B-A3B-Instruct）。**為何**：有限 GPU 下保留教師審查能力 + LoRA 可拆卸。
- **3.2 多階段 CoT 審查架構**：6 層（輸入/編排/檢索/模板/離線訓練/推論）+ 5 步 CoT（first_summary→first_code_review→linter→code_smell→total_summary）+ global_rule 8 準則（第 8 條為 RAG 動態注入）。**為何**：分步降單次 prompt 過載、逐步寫 .md 保證中途不失。
- **[深]** 每個設計選擇補一句 trade-off（為何 5 步而非單步、為何 RAG 注入為第 8 條、為何 LoRA 不合併）。**[銜]** 3.1 產出的學生模型正是 3.2 推論層所載入者，明示銜接。

### §4 Experiment Design

- **4.1 基準資料**：44 筆（GPT-5 + Copilot 生成、人工驗證），涵蓋語法/命名/設計模式/錯誤型態；**為何 44 筆**：覆蓋多維問題與標註成本的平衡。
- **4.2 評估方法**：LLM-as-a-Judge 架構 → CRSCORE++ 三維（comprehensiveness/conciseness/relevance）+ 自研 Our 五維（可讀/可維護/正確/跨評論覆蓋/簡潔）+ 人工交叉驗證。獨立評審（GPT-5）避免自評偏差。
- **4.3 RQ1–RQ4 與系統配置**。
- **[深]** 點明評估的**公正性設計**（評審獨立於審查模型、人工交叉驗證自動分），這是可信度的論證而非流程描述。**[構]** RQ↔表格對應在此先建立，§5 直接回填。

### §5 Results & Analysis

- **5.1 RQ1–RQ4 結果**：表 1（CRSCORE++ 0.86 vs 0.67 等）、表 2（LLM 評分消融）、表 3（人工評分）。
- **5.2 結果分析**：**為何**多階段提示詞 +34 為主導、LoRA 微調 +2 為邊際；自動評分與人工評分之趨勢一致性（除 Readability 略低）。
- **[深]** 從「報數字」升級為「數字說明什麼」——多階段提示詞的邊際貢獻遠大於微調，意味著「審查流程設計 > 模型微調」這一可推廣的結論。**[銜]** 每個 RQ 的結果段回指 §4 的對應表格與 §1.4 的對應貢獻。
