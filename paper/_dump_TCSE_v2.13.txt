[0017] <Normal 10.0pt> 大型語言模型於自然語言與程式語言理解上的突破，正快速滲透軟體工程之各個環節，從程式碼生成、缺陷偵測到文件撰寫均見其應用。在開發流程中，程式碼審查（Code Review）係指開發者相互檢視彼此之程式碼變更，不僅確保程式碼之品質、可維護性與一致性，更是團隊協作與知識傳遞的核心機制：潛在之邏輯錯誤、安全漏洞與效能問題得以在整合測試前被發現。然而，隨專案規模與迭代速度上升，純人工審查之時間與人力成本日增，逐漸成為制約開發效率與審查一致性之瓶頸，促使研究社群探索以人工智慧技術提升審查之自動化程度與可靠性。

[0025] <Normal 11.0pt> 二、相關研究
[0026] <Normal 10.0pt> 程式碼審查為軟體開發之關鍵環節，研究指出其可在整合測試前發現約 50% 至 70% 之缺陷 [13]，且審查意見之品質會直接影響後續程式修正之成效 [14]，故其自動化具高度價值。研究社群逐漸導入 LLM 作為輔助工具，以提升審查效率並協助開發者理解陌生領域 [2]，同時改善 Pull Request（開發者向主分支提出之合併申請）之審查延遲 [9]。另一方面，靜態分析工具（Static Analysis Tools, STAs）可在程式執行前檢測語法錯誤、潛在漏洞與風格問題並常整合於 CI/CD 流程，但其檢測能力受限於規則僵化，相當比例之問題無法涵蓋 [5][1]，且規則更新不易 [12]
[0027] <Normal 10.0pt> 大型語言模型以 Transformer（一種以自注意力機制為核心之深度學習網路）架構為基礎，透過大規模預訓練與下游微調理解自然語言與程式語言，已廣泛應用於程式輔助與內容生成，惟其仍面臨偏見與幻覺等問題，需搭配人工監督 [15]。在評估面，由於人工標註成本高昂，LLM-as-a-Judge 方法以大型語言模型自動對生成結果評分，其效果已被證實可與人類審查者相當 [13]，並能彌補 BLEU、ROUGE 等基於字串重疊之傳統指標無法反映語義品質之不足 [11]，相關研究亦提出多維度評估指標以提升評估之可靠性 [7][10]。
[0028] <Normal 10.0pt> 為抑制幻覺，檢索增強生成（RAG）藉由在生成前先自外部知識庫檢索相關內容並注入提示詞，可有效降低模型杜撰，已應用於安全建議與開發輔助系統 [8][19]。在模型輕量化方面，參數高效微調（Parameter-Efficient Fine-Tuning, PEFT）僅需調整少量參數即可降低訓練成本，其中 LoRA 透過低秩矩陣近似全參數更新，在極少參數下仍可逼近完整微調 [16]，QLoRA 則進一步結合低位元量化，使大型模型能於有限資源下完成微調並維持效能 [17][21]，知識蒸餾（Knowledge Distillation）則將大型教師模型之能力轉移至小型學生模型，於降低計算成本之同時維持
[0029] <Normal 10.0pt> 綜觀上述，四條技術線各自成熟，卻分別偏向單一面向，少見將其整合於同一審查框架者，形成一道整合缺口：LLM 審查多採單一提示詞架構，難以對複雜變更進行跨階段之逐步推理，致一致性受限，自動化評分雖可行，其與人類判斷之吻合度仍需交叉驗證，RAG 與輕量化技術雖已成熟，卻少與多階段審查流程整合，使領域規範與可落地部署難以同時兼得。本研究即以多階段 CoT 推理、RAG 知識注入、KD/QLoRA 輕量化與雙重 Judge（自動評分加人工評分）評估四者整合，一併補足語義理解、跨階段一致性、領域規範與評估可信度此一整合缺口。

[0040] <Normal 10.0pt> 提示模板層依當前審查步驟構建對應之英文提示詞範本，並由 global_rule 模板統一包入 7 條全域審查準則（涵蓋可讀性與一致性、命名、軟體工程原則、邏輯與正確性、效能與安全、文件與測試、評分與回饋風格）作為共同前綴。唯有當 RAG 層檢索到足夠相關之規則時，才再將其動態嵌入為條件式之第 8 條準則，亦即全域準則固定為 7 條靜態規則，第 8 條僅於檢索命中時出現。如此使外部領域知識不重新訓練模型即融入推理脈絡，並隨規則文件更新而即時調整。

[0069] <Normal 10.0pt> 於相同參數量之未微調模型上套用多階段提示詞，其 comprehensiveness 與 relevance 仍明顯優於 CRSCORE++，僅 conciseness 略遜（表一），顯示即使未經微調、在同等模型容量下，光是把審查流程結構化為多階段本身就能帶來實質之品質效益，可見多階段流程是整合框架中舉足輕重的一環，但如下節所示，它與微調、RAG 解決的是不同問題而仍需三者協同。
[0070] <Normal 10.0pt> 5.4 RQ3：提示詞與微調之邊際貢獻

[0077] <Normal 11.0pt> 六、結論
[0078] <Normal 10.0pt> 6.1 研究成果與貢獻
[0079] <Normal 10.0pt> 本研究針對將 LLM 用於程式碼審查時的三大問題——幻覺、輸出不穩定、缺乏領域規範——提出一套整合框架作為正面回應，以多階段 CoT 推理換取一致性、以 RAG 注入專案規範換取可解釋性、以知識蒸餾與 QLoRA 換取可落地之可維護性。三者整合，使審查同時具備此三性質。
[0080] <Normal 10.0pt> 經 LLM-as-a-Judge 與 CRSCORE++ 量化評估，於同一 judge 下，所提框架以 gemma-4-31B-it 與前代學生模型為基底之品質相當（表一：1.00 / 0.79 / 0.86 對 1.00 / 0.78 / 0.86），並由人工評分交叉驗證。消融分析則將此成效分解為各元件之貢獻：多階段流程之結構化設計為單項貢獻最大之一環（+34 分），LoRA 微調在有限資源下保留並穩定教師之審查能力（+2 分），RAG 則提供領域規範之接地，三者互補而缺一不可。本研究的核心發現不在於某一元件勝出，而在於唯有三者整合方能同時改善三性質、回應前述三大問題。
[0081] <Normal 10.0pt> 6.2 研究限制與未來工作
[0082] <Normal 10.0pt> 本研究仍有若干限制與發展方向。在資料集方面，目前以 44 筆、以 Python 為主之測試資料驗證，規模相對有限，未來可擴展至涵蓋多程式語言、多框架與真實開源專案之大規模資料集。在模型能力方面，本研究主要對單一學生模型微調，未來可探索多模型協作（multi-agent）機制，使各模型分別負責特定審查面向（如安全、效能、風格）。在評估範圍方面，§3.2 所述之框架雖已部署於 GitHub Actions CI/CD、MCP 與 pre-commit 等路徑實際運作，但其可替換之多後端推論的品質、成本與延遲比較仍屬未來工作而未於本文實測，反饋語料累積之效益、以及十七項擴充機制之端到端品質效益，亦均
