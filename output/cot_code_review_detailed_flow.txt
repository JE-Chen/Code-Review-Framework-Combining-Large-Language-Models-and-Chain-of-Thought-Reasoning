本架構描述 CoT (Chain-of-Thought) 程式碼審查管線的細部流程。當原始碼輸入後，先經過 Query Embedding 並於 FAISS 向量索引中檢索相關規則，再將擷取到的規則注入到 Global Template，使後續每一個審查步驟都帶有專案規範與領域知識。注入完成後，系統並行執行四個子任務：Step 1 Summary 產出變更摘要、影響範圍與風險評估；Step 2 Code Review 聚焦可讀性、命名規範與邏輯錯誤；Step 3 Linter 以 rule_id、severity、line、suggestion 結構化標註問題；Step 4 Code Smell 則從 Type、Location、Priority 與 Refactoring 角度找出壞味道。四個步驟結果匯入 Step 5 Total Summary，由模型綜合所有觀察並產出 Merge 建議與 Follow-up 行動，其中 Linter 與 Code Smell 結果亦會直接輸入 Judge。最終由 LLM Judge 依 Readability、Constructiveness、Correctness、Coverage & Extractability、Comprehensiveness 五個面向各打 1–100 分，產出 Final Score 與最終建議。此設計透過拆解、平行化與規則注入，提高長段程式碼審查的覆蓋率與可解釋性。
